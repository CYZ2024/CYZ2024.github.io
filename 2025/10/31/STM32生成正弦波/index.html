STM32生成正弦波

发表于 2025-10-31 分类于 学习笔记

STM32 双核项目实验报告：基于 PWM+DMA 的正弦波发生器与 ADC 采样系统

一、实验目的

本次实验使用两块 STM32 微控制器搭建了一个完整的信号生成与采集系统，主要想实现以下功能：

  • 信号发生器部分 (STM32F411RET6)：通过 PWM+DMA 的方式生成可调频率的模拟正弦波信号
  • 数据采集部分 (STM32F103C8T6)：用 ADC+TIM+DMA 对生成的信号进行采样，并通过串口把数据传到电脑上分析

二、方案设计

2.1 信号发生器方案 (STM32F411)

一开始我想用 DAC 来生成正弦波，查了下手册发现 F411 和 F103 都没有集成 DAC 模块，只能换个思路。

最终采用的方案：PWM + RC 低通滤波器

基本原理是这样的：用高频 PWM（我设置的是 50 kHz）动态改变占空比，让占空比按照正弦规律变化。这个 PWM 信号经过 RC 滤波器之后，高频成分被滤掉，输出的电压就是占空比对应的平均值，这样就能得到一个模拟的正弦波了。

2.2 数据采集方案 (STM32F103)

  • 采样触发：用 TIM3 定时器作为 ADC 的外部触发源，这样可以保证采样频率稳定，满足采样定理的要求
  • 数据搬运：配置 DMA 自动把 ADC 转换结果搬到内存缓冲区，不用 CPU 参与
  • 数据传输：通过 UART 串口把采集到的数据发送到电脑

三、信号发生器的实现 (F411)

3.1 定时器 PWM 配置

我把 TIM3 配置成 PWM 模式，系统时钟是 72 MHz。

  • PWM 频率计算：

  • 比较寄存器 (CCR)：这个寄存器的值决定占空比，范围是 0 到 1439，是实现正弦波的关键

3.2 DMA 配置

为了自动更新 CCR 的值，我提前在内存里生成了一个正弦查找表。

  • 触发方式：定时器更新事件触发 DMA 传输
  • DMA 工作模式：
    • 循环模式，这样波形可以连续输出
    • 传输方向：内存到外设
    • 数据宽度：16 位半字，跟 CCR 寄存器匹配

3.3 遇到的问题一：内存不够用

问题描述：编译的时候 Keil 报错L6406E: No space in execution regions，说是内存空间不足。

原因分析：仔细检查了 CubeMX 的配置，发现不小心开启了 USB_DEVICE 中间件，它默认要分配 32 MB 的 RAM，但 F411 总共才 128 KB 的 RAM。

解决办法：在 CubeMX 里把 USB 中间件禁用掉，重新生成代码就好了。

3.4 功能验证

为了验证 PWM+DMA 系统是否正常工作，我把正弦波的频率调到 1 Hz（就是让 DMA 传输特别慢）。

然后在 PWM 输出引脚接了个 LED（串了限流电阻），观察到 LED 出现了平滑的”呼吸”效果，亮度按正弦规律变化。这说明 PWM 和 DMA 部分基本没问题。

  • 关闭连续转换模式，确保每次采样都是由定时器精确触发的

4.2 DMA 和串口配置

DMA 配置：

  • 传输方向：外设到内存
  • 目标缓冲区：uint16_t adc_buffer[1000]
  • 工作模式：循环模式

串口配置：波特率设置为 115200，用来把数据发送到电脑

4.3 遇到的问题二：采集不到数据

问题描述：串口自检正常（上电会打印”Hello!“），但是完全收不到 ADC 的采样数据。

问题分析：主循环在正常运行，说明程序没死。但adc_data_ready_flag标志位一直是 0，说明HAL_ADC_ConvCpltCallback这个回调函数从来没被调用过。

排查过程：

  1. 检查了 NVIC 中断配置，DMA1 Channel 1 和 ADC1 的中断都已经使能
  2. 检查了stm32f1xx_it.c文件，确认中断处理函数里正确调用了HAL_DMA_IRQHandler(&hdma_adc1)
  3. 检查了adc.c，DMA 句柄定义和声明都没问题

问题定位：最后发现是 TIM3 的 ARR 寄存器没配置对，还保留着默认值（0 或者 65535），导致 ADC 触发频率完全不对，整个 DMA 流程就乱套了。

解决方法：把 F103 的 TIM3 的 ARR 寄存器改成 99，保证 5 kHz 的触发频率。改完之后系统就开始正常输出数据了。

五、波形失真问题的解决

5.1 发现的问题

把采集到的数据导入 Excel 画图之后，发现波形的顶部和底部都是平的，出现了明显的削波失真。

5.2 第一个猜测：PWM 输出饱和

F411 的 CCR 值范围是[0, 1439]。

当 CCR 接近 1439（100%占空比）或者 0（0%占空比）的时候，PWM 输出会被钳位在 3.3V 或 0V，RC 滤波器的电容充放电到了电源的极限，没法再继续变化了，所以会出现削波。

5.3 真正的原因：ADC 量程限制

我尝试把 PWM 的范围压缩到[145, 1295]，结果波形底部恢复正常了，但是顶部还是平的，ADC 读数一直卡在 2150 左右。

结论：F103 的 ADC 虽然理论上是 12 位的（0-4095），但实际有效测量范围只有 0 到 2150 左右。这可能跟板子上 VDDA 模拟电源电压偏低或者电路的非线性特性有关。

六、实验总结

6.1 技术收获

这次实验让我真正理解了 TIM、DMA、ADC、UART 这些外设是怎么配合工作的，特别是触发链和数据流的设计。之前只是在课本上学过理论，这次实际动手才知道有这么多细节要注意。

6.2 调试经验

  1. 模块化验证很重要：先用串口自检确认基本通信正常，然后再排查 ADC 和 DMA 的问题，这样能快速定位故障点
  2. 中断配置要仔细检查：NVIC 使能、中断处理函数、外设句柄，每一步都不能出错
  3. 时序配置是关键：TIM3 的 ARR 寄存器配置错误导致整个采集系统失效，说明硬件时序必须精确

6.3 理论与实际的差距

最大的体会是理论参数和实际情况可能有很大差别。比如 ADC 的理论量程是 0-4095，但实际只能用到 2150。这次实验让我明白了，做硬件系统不能光靠理论计算，必须要根据实测数据来调整参数。

6.4 数据驱动的方法

通过采集数据、分析波形、反推参数，最后解决了削波问题。这种基于实测数据进行系统标定的方法在实际工程中应该很常用，这次算是有了一个比较深刻的体会。
